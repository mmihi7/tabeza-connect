AGENT ADVISORY
Receipt Parsing Architecture
AI-First with Venue-Level Regex Caching

Purpose
This advisory documents architectural decisions for the Tabeza receipt parsing system. It supersedes any prior assumptions about parsing rules, rejection logic, or the role of Tabeza relative to the POS system. Agents implementing the parser, relay route, or Captain's Orders pipeline must read this document before writing code.



1.  Tabeza's Role — The Non-Negotiable Constraint
?  FOUNDATIONAL RULE — Read Before Implementing Anything
The POS system is the source of truth. Always. Tabeza is a convenience layer.

The POS always runs. It always prints. It always records the sale.
If Tabeza is down, slow, or fails to parse — the bar operates perfectly.
The customer simply does not receive the optional digital convenience features.

Tabeza must NEVER block, delay, reject, or interfere with a print job for any reason.


This constraint has direct implications for every parsing decision:

•Never reject a receipt because parsing failed.
•Never block the print relay waiting for an AI response.
•Never drop a receipt record because items could not be extracted.
•Always accept, always store, always make failures visible — not silent.



2.  Why AI-First (Not Regex-First)
The Tabeza deployment spans hundreds of POS systems across Kenya and the region. These include international brands (Oracle MICROS, Square, Lightspeed), local vendors, custom-built systems, and POS software with Swahili-language labels and non-standard receipt templates.
Regex is effective when you know the format in advance. Against hundreds of unknown, undocumented POS formats it degrades unpredictably — some receipts parse perfectly, others produce silent partial failures or garbage output with no indication of which outcome occurred.
?  Architecture Decision: AI First, Regex as Safety Net
AI handles the long tail of unknown formats without enumeration.
Regex handles cases where AI is unavailable, rate-limited, or returns low-confidence results.
The regex library built to date remains valuable — it is the fallback and the zero-cost path
for formats already seen. It is not the primary parser.




3.  Venue-Level Format Caching — The Core Optimisation
This is the most important architectural decision in this document. It directly controls cost, latency, and reliability at scale.
3.1  The Insight
A venue's POS system prints the same format thousands of times. Once the AI has successfully parsed a receipt from a venue, that format is known. Sending every subsequent receipt from that same venue through the AI is wasteful — it costs money, adds latency, and introduces an unnecessary dependency on an external API.
3.2  The Caching Strategy
Step	Trigger	Action
1	First receipt from a venue (or no cached pattern)	Send to AI parser. On success, extract the structural pattern and store it against the venue ID.
2	Subsequent receipts from the same venue	Use cached regex pattern. No AI call. Instant, free, no external dependency.
3	Cached regex returns low confidence or fails	Invalidate the cached pattern. Fall back to AI. The POS may have changed its template.
4	AI also fails (timeout, rate limit, bad response)	Fall back to general regex library. Store raw receipt. Flag as low confidence. Never discard.

3.3  What "Extract the Pattern" Means
When the AI successfully parses a receipt, the system should derive a lightweight regex signature from the successful parse — specifically the structural markers: column positions, label strings, currency format, date format, and separator characters. This derived pattern is stored in the database against the venue ID.
This is not asking the AI to write regex. It is observing the structure of a receipt the AI successfully parsed and encoding that structure as a pattern for future fast matching. The AI does the hard work once; regex does the cheap work forever after.



4.  Parsing Failure Policy
The following rules govern what happens when parsing produces incomplete or no results. They replace any prior "reject on parse failure" logic.

Parse Result	Confidence	System Behaviour
Items + total extracted	High	Create full tab. Show itemised view to customer.
Total only, no items	Medium	Create tab with total. Show summary view. Flag for staff review.
Partial items, total missing	Medium	Create tab with available items. Flag for staff review.
Nothing extracted	Low	Store raw receipt record. Mark status: pending. Visible to staff. Never discarded.

Why failures must be visible, not silent
A bar manager who sees a flagged receipt in Captain's Orders can investigate.
A bar manager who does not know a receipt was silently dropped cannot.
Revenue loss from silent discards is invisible by definition — the worst kind of bug.




5.  Where to Enforce Parsing Logic
Printer Service (index.js) — Physical Print Only
The printer service has one job: relay the print data to the cloud. It must always complete this job regardless of parsing outcome. Do not add parsing rejection logic, confidence checks, or AI calls to the printer service. It runs on a Windows laptop at the venue; it should be as simple and reliable as possible.
Relay API Route — Parsing Trigger
The relay route is the correct place to initiate parsing. It receives the raw receipt data, triggers the AI parser (or cached regex if available), stores the result with the appropriate confidence level, and returns a job ID. Parsing is asynchronous where possible — the relay route should not make the printer service wait for an AI response.
Captain's Orders UI — Confidence Display
The UI should reflect confidence levels visually. High-confidence tabs display normally. Medium-confidence tabs show a review indicator. Low-confidence tabs (raw receipt only) appear in a separate holding queue visible to staff. The customer-facing display degrades gracefully — a rich itemised tab when available, a simple total view when not.



6.  Anti-Patterns — What Not to Build
•Do not add parsing logic to the printer service.  It belongs in the cloud.
•Do not silently discard receipts.  Every receipt must land somewhere in the system.
•Do not block the relay on AI response time.  Accept the receipt immediately; parse asynchronously.
•Do not treat Tabeza as a POS replacement.  It is a convenience layer. The POS is always right.
•Do not route all receipts through AI indefinitely.  Build the venue cache. Cost compounds at scale.
